# ARC — 프로젝트 컨텍스트

## 서비스 개요

ARC는 **포트폴리오 아카이빙 및 AI 연계 자동화 플랫폼**이다.  
이름은 Story Arc에서 착안했으며, 한 사람의 경력과 경험 조각들을 모아 하나의 내러티브로 만들어주는 서비스다. Archive의 약어이고 방주(Ark)를 뜻하기도 한다.

## 핵심 철학

- 이력은 단순한 스펙 목록이 아니다. 사용자가 어떤 동기로 활동을 선택했는지, 어떤 고민을 했는지, 무엇을 배웠는지와 같은 **정성적 경험의 총체**다.
- ARC는 이 정성적 데이터를 수집하고 연결하는 데 집중한다. 이것이 링커리어, LinkedIn 등 타 플랫폼과의 핵심 차별점이다.
- **기록의 허들을 낮추는 것이 최우선**이다. 사용자가 기록을 많이 쌓을수록 분석 품질이 올라가는 구조이므로, UX는 항상 입력 편의성을 중심으로 판단한다.
- 주요 타겟은 **진로가 확정되지 않은 대학생**이다. 이들은 불안감과 강박에서 비롯된 소비를 하기 때문에, ARC는 단순한 도구가 아니라 안정감을 주는 플랫폼이어야 한다.

## AI 활용 원칙

1. **AI 기능은 백엔드 팀이 별도 구현**한다. 프론트엔드는 백엔드가 제공하는 API 엔드포인트를 호출하는 방식으로 연동한다. 프론트에서 AI 모델을 직접 호출하지 않는다.
2. **AI 챗봇 기반 경험 추출**은 단순 편의 기능이 아니라 분석 인프라다. 정성 데이터를 얼마나 잘 받아내느냐가 전체 분석 품질을 결정한다.
3. 목표가 없는 사용자에게는 먼저 "당신은 이런 사람 같네요"를 제시해 흥미를 유도한 뒤, 자연스럽게 목표 설정으로 이어지도록 설계한다.
4. **내러티브 엔진**이 ARC의 핵심 차별점이다. 단순 키워드 태깅을 넘어, 수업-활동-인턴의 흐름을 고려해 사용자 스스로도 몰랐던 스토리라인을 제안한다.

## 비즈니스 모델

- **초기**: 구독료 기반. 무료 플랜은 추출 문서에 워터마크 적용, Pro 플랜에서 제거.
- **이후**: 축적된 사용자 데이터를 활용한 인포 사업 (채용 시장 연계 등).

## 개발 시 주의사항

- 컴포넌트 작성 시 **적응형 웹** 기준으로 Tailwind 클래스 작성
- 페이지 간 상태는 최대한 URL(searchParams)로 관리, 전역 상태는 최소화
- AI 관련 기능은 백엔드 API 엔드포인트로 분리되어 있음. 프론트는 fetch/axios로 호출만 한다
- 워터마크 로직은 Pro 여부를 서버에서 검증 후 적용

## Git 워크플로우

### 브랜치 전략 (GitHub Flow)

```
main
 └── feat/기능명       새 기능 개발
 └── fix/버그명        버그 수정
 └── docs/문서명       문서 작업
 └── refactor/대상     코드 리팩토링
 └── chore/작업명      설정, 의존성 등 기타 작업
 └── hotfix/이슈명     긴급 수정
```

**브랜치 네이밍 규칙**: `타입/간결한-설명` (영문 소문자, 하이픈 구분)

```bash
# 작업 시작
git checkout main && git pull origin main
git checkout -b feat/archive-input-form

# 작업 후 PR
git push origin feat/archive-input-form
# GitHub에서 PR 생성 → 리뷰 → merge → 브랜치 삭제
```

### ⛔ main 브랜치 직접 커밋 금지

```bash
# 절대 금지
git push origin main
git commit -m "..." # main 브랜치에서
```
